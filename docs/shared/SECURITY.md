# دليل الأمان — مشروع رقية التعطيل

هذا المشروع يتبع **AI Vibe Coding - Security Playbook** (24 قاعدة). القواعد مضمّنة في `.cursor/rules/security-playbook.mdc` لتوجيه التطوير.

## ما تم تطبيقه حالياً (تطبيق Flutter فقط)

| القاعدة | التطبيق |
|--------|---------|
| **10** عدم ظهور سجلات في الإنتاج | جميع استدعاءات `debugPrint` محاطة بـ `kDebugMode` فلا تُنفَّذ في بناء الإصدار. |
| **03** حماية المفاتيح | لا توجد مفاتيح API في الكود؛ عند إضافة باكند استخدم متغيرات بيئة أو `--dart-define`. |
| **16** تخزين المستخدم | `SharedPreferences` معزولة لكل تطبيق = كل مستخدم يرى بياناته فقط. |
| **05–07** الحزم والإصدارات | قبل إضافة حزمة: تحقق من الأمان و`dart pub outdated`؛ بعد التحديث راجع التبعيات. |

## عند إضافة باكند أو واجهات API

- **01–03**: استخدم JWT مع انتهاء صلاحية ≤ 7 أيام وتدوير refresh، ولا تبني نظام تسجيل دخول بالذكاء الاصطناعي — استخدم Clerk / Supabase / Auth0. احفظ المفاتيح في `process.env` فقط.
- **04**: دوّر الأسرار كل 90 يوم على الأقل.
- **08**: تعقيم المدخلات واستعلامات مُعلَّمة فقط.
- **09**: تفعيل Row-Level Security في قاعدة البيانات من اليوم الأول.
- **11–12**: CORS مقيد بالنطاق المسموح، والـ redirect URLs من قائمة مسموحة فقط.
- **13**: مصادقة وحد معدل (rate limiting) على كل endpoint.
- **14–15**: حد أقصى لتكلفة واجهات الذكاء الاصطناعي، وحماية من DDoS (مثل Cloudflare).
- **17–18**: التحقق من الرفعات بالتوقيع وليس بالامتداد؛ التحقق من توقيع الـ webhooks قبل معالجة الدفع.
- **19**: فرض الصلاحيات من جانب الخادم — فحوصات الواجهة ليست أماناً.
- **20–21**: تسجيل الإجراءات الحرجة (حذف، تغيير أدوار، مدفوعات، تصدير)؛ بناء تدفق حذف حساب حقيقي.
- **22–24**: نسخ احتياطية آلية ومختبرة؛ فصل بيئة الاختبار عن الإنتاج؛ عدم تعريض الـ webhooks لأنظمة حقيقية في بيئة الاختبار.
